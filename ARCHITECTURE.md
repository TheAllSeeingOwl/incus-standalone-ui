# Incus Manager — Architecture & Developer Reference

A Tauri v2 desktop application that embeds the [incus-ui-canonical](https://github.com/zabbly/incus-ui-canonical)
web UI inside a native window, with a local Rust proxy handling TLS and mTLS to the Incus server.

## Overview

```
┌─────────────────────────────────────────────────┐
│  Tauri window  (tauri://localhost)               │
│                                                  │
│  ┌──────────┐  ┌──────────────────────────────┐ │
│  │ Settings │  │  <iframe>                    │ │
│  │ sidebar  │  │  http://127.0.0.1:<port>/    │ │
│  │ (React)  │  │  <secret>/ui/                │ │
│  │          │  │                              │ │
│  │          │  │  incus-ui-canonical SPA      │ │
│  └──────────┘  └──────────────────────────────┘ │
└─────────────────────────────────────────────────┘
          │ Tauri IPC (get_proxy_port,
          │ save_settings, …)
          ▼
┌─────────────────────────────────────────────────┐
│  Rust backend                                    │
│                                                  │
│  axum proxy  127.0.0.1:<random port>             │
│  ├── /<secret>/ui/*  →  embedded SPA assets      │
│  ├── /<secret>/1.0/* →  Incus REST API (mTLS)    │
│  ├── /<secret>/oidc/*                            │
│  └── /<secret>/documentation/*                  │
└───────────────────────┬─────────────────────────┘
                        │ HTTPS / mTLS
                        ▼
              Incus server :8443
```

The app never opens a browser. The Incus UI runs entirely inside the Tauri
webview. The Rust proxy handles all TLS complexity, including self-signed
certificates and mutual TLS client authentication.

## How It Works

### Startup sequence

1. Tauri launches and runs the `setup` closure in `lib.rs`
2. Settings are loaded from the persistent store (`incus-settings.json`)
3. `start_proxy()` is called:
   - Binds a TCP listener on `127.0.0.1:0` (OS assigns a free port)
   - Generates a random 32-byte hex secret
   - Builds a `reqwest::Client` with the configured TLS settings
   - Spawns an axum server as a tokio task
   - Returns the port, secret, and shared proxy state handle
4. The main window is created pointing at `tauri://localhost` (our React shell)
5. The system tray icon is set up
6. On first run (no host configured), the settings sidebar is opened automatically

### Window architecture

There is a single Tauri window (`"main"`). It loads our React shell from the
Tauri custom protocol (`tauri://localhost`), which serves the compiled `dist/`
assets embedded in the binary.

The shell renders two things side by side:
- A collapsible settings sidebar (React component, talks to Rust over IPC)
- An `<iframe>` that loads the Incus UI from the local proxy

The settings sidebar is the only part that ever calls Tauri IPC commands.
The iframe (Incus UI) cannot access `window.__TAURI__` — it is sandboxed to
its own origin and has no IPC access.

The tray menu has three items:
- **Open** — shows/hides the main window
- **Settings** — calls `window.__openSettings()` in the shell to open the sidebar
- **Quit** — exits the app

Closing the window hides it to the tray rather than quitting.

## Proxy Design

The proxy (`src-tauri/src/proxy.rs`) is an axum HTTP server running on
`127.0.0.1` on a random port chosen by the OS at startup.

### Routing

| Path pattern | Handler |
|---|---|
| `/<secret>/1.0` | Forward to Incus REST API |
| `/<secret>/1.0/*` | Forward to Incus REST API |
| `/<secret>/oidc/*` | Forward to Incus (OIDC auth) |
| `/<secret>/documentation/*` | Forward to Incus |
| `/<secret>/ui/*` | Serve embedded SPA static files |
| everything else | 403 (secret missing or wrong) |

### Static file serving

The incus-ui-canonical SPA is embedded at compile time using `include_dir!`.
This means the entire UI (several MB including the Monaco editor) lives inside
the binary — no filesystem access needed at runtime, no tampering possible.

Cache headers:
- `assets/*` (content-hashed filenames) → `Cache-Control: public, max-age=31536000, immutable`
- `index.html` and other entry points → `Cache-Control: no-cache, must-revalidate`

Unknown paths fall back to `index.html` to support client-side routing in the SPA.

### API proxying

HTTP requests are streamed in both directions — neither the request body nor
the response body is fully buffered in memory. This matters for:
- File/image uploads to Incus
- Large responses (container exports, log streams)
- Server-sent events from the Incus events endpoint

CORS headers are added to all API responses so the iframe origin can read them.
`OPTIONS` preflight requests are answered locally without forwarding to Incus.

### WebSocket relay

The Incus UI uses WebSocket connections for container console access and exec
sessions. The proxy detects `Upgrade: websocket` requests and relays them
end-to-end using `tokio-tungstenite` with the same TLS configuration as the
HTTP client. Both directions run concurrently via `tokio::select!`.

### State management

Proxy configuration (host, port, certs) is stored in an `ArcSwap<ProxyStateInner>`.
`ArcSwap` provides lock-free reads: each request does a single atomic pointer
load with no blocking. When settings change, `save_settings` atomically swaps
in a new `ProxyStateInner` with a freshly built `reqwest::Client`. In-flight
requests complete with the old client; new requests immediately use the new one.

## Settings & Configuration

Settings are persisted by `tauri-plugin-store` to a JSON file at:

| Platform | Path |
|---|---|
| Linux | `~/.local/share/app.incus.manager/incus-settings.json` |
| macOS | `~/Library/Application Support/app.incus.manager/incus-settings.json` |
| Windows | `%APPDATA%\app.incus.manager\incus-settings.json` |

### Fields

| Field | Type | Default | Description |
|---|---|---|---|
| `host` | string | `localhost` | Incus server hostname or IP |
| `port` | number | `8443` | Incus HTTPS port |
| `acceptInvalidCerts` | bool | `false` | Skip TLS certificate verification |
| `caCertPath` | string? | null | Path to a PEM CA certificate to trust |
| `clientCertPath` | string? | null | Path to a PEM client certificate (mTLS) |
| `clientKeyPath` | string? | null | Path to the corresponding PEM private key |

### Applying settings

Clicking **Apply** in the settings sidebar calls `save_settings` over IPC.
The Rust handler:
1. Persists the new values to the store
2. Builds a new `reqwest::Client` with the updated TLS configuration
3. Atomically swaps it into the proxy state via `ArcSwap::store`
4. The React shell reloads the iframe src, which reconnects to Incus

No proxy restart or window reload is required.

## Security Model

### Threat model

The app runs on a personal workstation. It holds a TLS client certificate that
grants administrative access to an Incus server. The threats we consider are:

- A web page in another browser tab trying to abuse the local proxy
- Another process on the machine trying to use the proxy as a privilege escalation vector
- DNS rebinding attacks from malicious websites

### Defences

#### Host header validation

Every request is checked against a middleware that rejects any `Host` header
that is not exactly `127.0.0.1:<port>` or `localhost:<port>`. This blocks
DNS rebinding attacks, where a malicious page remaps a domain to `127.0.0.1`
to reach the proxy from a browser tab.

#### Iframe isolation

The Incus UI runs in an `<iframe>` with origin `http://127.0.0.1:<port>`.
Tauri's `window.__TAURI__` IPC bridge is only injected into the top-level
frame (`tauri://localhost`). The iframe cannot call any Tauri commands —
it cannot read settings, cannot modify the proxy configuration, and cannot
access the secret token.

#### Localhost-only binding

The proxy binds to `127.0.0.1` (not `0.0.0.0`). It is unreachable from
the network by design.

#### Compile-time asset embedding

The SPA is embedded via `include_dir!` at compile time. There are no runtime
file reads for serving the UI, so the assets cannot be tampered with after the
binary is built.

### Remaining considerations

| Topic | Notes |
|---|---|
| **Local process access** | Any same-user local process can reach the proxy. This is acceptable — such a process can already read the cert files directly from disk |
| **`acceptInvalidCerts`** | Disables all TLS verification. A visual warning is shown. Only use on a trusted local network — a MITM can intercept all Incus API traffic |
| **Client key file permissions** | The key file should be `chmod 600` (readable only by owner). The app does not enforce this |
| **App data directory** | Contains cert *paths* (not the key material itself) in a plain JSON file. Keep `~/.local/share/app.incus.manager/` as `700` |
| **Supply chain** | The SPA is a third-party git submodule. Review upstream changes before updating the submodule, especially given it has privileged access to Incus |

## Performance Decisions

| Decision | Rationale |
|---|---|
| **Streaming proxy bodies** | Neither request nor response is buffered in full. Large payloads (exports, uploads, log streams) flow through with constant memory use |
| **`ArcSwap` for proxy state** | Lock-free reads on every request. No blocking even while settings are being saved |
| **`Cache-Control: immutable` on assets** | Content-hashed SPA assets are cached by the webview forever. Reconnecting after a settings change does not re-download megabytes of JS |
| **`OPTIONS` preflight short-circuit** | Preflight requests are answered locally in microseconds, not forwarded to Incus |
| **Connection pool + TCP keepalive** | `pool_max_idle_per_host(8)` and `tcp_keepalive(20s)` keep connections to Incus warm so the first request after idle does not pay a full TCP+TLS handshake |
| **`include_dir!` embedding** | SPA served from memory (no disk I/O per request). Startup is fast because there is no unpacking step |
| **HTTP/2 via ALPN** | `reqwest` negotiates HTTP/2 automatically if Incus supports it, multiplexing parallel API calls over a single connection. `http2_prior_knowledge` is intentionally NOT used — it forces cleartext HTTP/2 and breaks TLS connections |

## Development Environment

The development environment is managed by [devenv](https://devenv.sh) and
activated automatically by [direnv](https://direnv.net) when you enter the
directory.

### First-time setup

```bash
# 1. Allow direnv (only needed once)
direnv allow

# 2. Initialise the submodule
git submodule update --init --recursive

# 3. Install JS dependencies
yarn install
yarn --cwd incus-ui-canonical install --frozen-lockfile
```

### Running in development

```bash
cargo tauri dev
```

This will:
1. Start the Vite dev server for the shell (port 5173)
2. Build the Rust backend with hot-reload
3. Open the Tauri window

> **Note:** In dev mode the SPA is built from the last `yarn --cwd incus-ui-canonical build`
> output embedded at compile time. Re-run that command if you update the submodule.

### What devenv provides

| Tool | Version |
|---|---|
| Rust / cargo | stable (from nixpkgs) |
| cargo-tauri | 2.x |
| Node.js | 22.x |
| Yarn | 1.x |
| webkit2gtk | 4.1 |
| libayatana-appindicator | system tray support |
| pkg-config + all GTK/GLib libs | Tauri Linux build deps |

`LD_LIBRARY_PATH` is set automatically so dynamic libraries are found at
runtime without needing to install anything system-wide.

## Building

### Development build

```bash
cargo tauri dev
```

### Release build

```bash
# Build the SPA first (only needed if submodule changed)
yarn --cwd incus-ui-canonical build

# Build the Settings shell
yarn build

# Build and package the Tauri app
cargo tauri build
```

Output bundles are in `src-tauri/target/release/bundle/`.

### Build pipeline (`beforeBuildCommand`)

When running `cargo tauri build`, Tauri automatically runs:

```
yarn --cwd incus-ui-canonical install --frozen-lockfile
  && yarn --cwd incus-ui-canonical build
  && yarn build
```

This ensures the SPA and shell are always built before the Rust compile step
that embeds them via `include_dir!`.

## Known Limitations

| Limitation | Details |
|---|---|
| **Binary size** | The Monaco editor (used by incus-ui-canonical for cloud-init editing) adds ~25 MB to the embedded assets. This is inherited from the upstream SPA and cannot be reduced without patching it |
| **Single Incus server** | The app connects to one Incus server at a time. Switching servers requires changing settings and reconnecting |
| **No certificate UI feedback** | If the client certificate is rejected by Incus, the iframe shows a generic connection error rather than a specific TLS error message. Check the terminal output for `proxy error:` lines |
| **Proxy port changes on restart** | The proxy binds to a random port on each launch. This is intentional (avoids port conflicts) but means bookmarking the proxy URL directly won't work across restarts |
| **Linux only tested** | The devenv is configured for Linux (webkit2gtk). macOS and Windows are supported by Tauri but require adjusting the system deps in devenv.nix and testing |
